# 构建第一个现代应用程序 - UI 设计

在前面的学习中，你已经知道 AI 可以帮你快速生成代码。但很多同学第一次做应用时，最容易遇到的问题不是“代码跑不起来”，而是：

- 页面看起来像临时拼出来的
- 按钮、卡片、表单都有，但不知道哪里不对
- AI 每次生成的界面风格不稳定
- 页面能用，但不像一个真正的产品

这节课要解决的就是这个问题：**如何用一套简单、可复用的 UI 设计方法，让 AI 帮你做出更像现代应用的界面。**

你不需要先成为专业设计师。你只需要学会判断一个页面的结构、层级、状态和一致性，然后把这些要求清楚地告诉 AI。

## 你将学到什么

完成本节后，你应该能做到：

1. 看懂一个现代 Web 应用页面的基本结构
2. 区分页面里的主操作、次操作和辅助信息
3. 设计常见的按钮、表单、卡片、导航和空状态
4. 让 AI 按统一的视觉规范生成页面
5. 用检查清单判断一个 UI 是否已经达到可演示水平

本节不追求复杂的视觉创意，而是追求一个更重要的目标：**让你的第一个应用看起来清楚、稳定、可信。**

## 1. 现代应用页面通常由什么组成

一个应用页面看起来再复杂，本质上也可以拆成几类基础模块：

| 模块 | 作用 | 常见位置 |
| --- | --- | --- |
| 导航 | 告诉用户在哪里，可以去哪 | 顶部导航、侧边栏 |
| 标题区 | 告诉用户当前页面做什么 | 页面顶部 |
| 操作区 | 放主要按钮、筛选、搜索 | 标题区右侧或内容上方 |
| 内容区 | 展示核心数据或任务 | 页面中心 |
| 反馈区 | 告诉用户当前状态 | 弹窗、Toast、空状态、加载状态 |

很多新手页面不好看，不是因为颜色不高级，而是因为这些模块混在一起：

- 标题不像标题
- 按钮没有主次
- 信息密度忽高忽低
- 所有东西都挤在页面中间

做 UI 的第一步，是先把页面分层，而不是先选颜色。

## 2. 先画页面骨架，再让 AI 生成代码

在让 AI 写页面之前，先用文字写出页面骨架。比如你要做一个 AI 文案生成工具，可以先这样拆：

```text
页面：AI 文案生成工作台

顶部：
- 产品名称
- 当前用户信息
- 设置入口

主体：
- 左侧：生成参数表单
- 右侧：生成结果预览

表单字段：
- 文案类型
- 产品名称
- 目标用户
- 语气风格
- 生成按钮

结果区：
- 加载状态
- 生成结果
- 复制按钮
- 重新生成按钮
```

这个骨架比“帮我做一个好看的页面”更有用。因为 AI 最怕的不是任务复杂，而是你没有告诉它页面里到底有什么。

## 3. 信息层级：让用户一眼知道重点

页面里的信息不是平等的。你要让用户一眼看出：

1. 当前页面是什么
2. 用户现在应该做什么
3. 哪些信息只是辅助说明

一个简单的层级规则是：

| 层级 | 用途 | 设计方式 |
| --- | --- | --- |
| 一级 | 页面标题、核心任务 | 更大字号、更高字重 |
| 二级 | 分区标题、重要数据 | 中等字号、清晰间距 |
| 三级 | 说明文字、辅助信息 | 较小字号、低对比度 |
| 操作 | 按钮、链接、输入框 | 明确边界和状态 |

例如：

```text
不好：
AI文案生成
输入信息
开始
结果

更好：
AI 文案生成工作台
输入产品信息，生成适合投放的营销文案

[文案类型] [产品名称] [目标用户]
[生成文案]

生成结果
这里展示 AI 返回的候选文案，可以复制或重新生成
```

第二种写法不只是文字更多，而是每一层都承担了清晰职责。

## 4. 按钮设计：主按钮只能有一个

很多页面显得混乱，是因为所有按钮都在抢注意力。

一个页面或一个区域里，通常只应该有一个主按钮。其他按钮应该降级为次按钮、文字按钮或图标按钮。

| 类型 | 使用场景 | 示例 |
| --- | --- | --- |
| 主按钮 | 页面最希望用户完成的动作 | 生成文案、保存、提交 |
| 次按钮 | 重要但不是最主要的动作 | 预览、取消、导出 |
| 文字按钮 | 低风险辅助操作 | 查看详情、清空 |
| 危险按钮 | 删除、重置等不可逆操作 | 删除项目 |

例子：

```text
主按钮：生成文案
次按钮：保存草稿
文字按钮：清空输入
危险按钮：删除历史记录
```

不要把所有按钮都做成高饱和颜色。颜色越多，用户越不知道该点哪里。

## 5. 表单设计：让用户知道怎么填

AI 应用里经常有大量表单，比如生成参数、用户资料、任务配置。表单设计的核心不是漂亮，而是降低用户填写成本。

一个好表单通常包含：

- 清楚的字段名称
- 必要的占位提示
- 默认值
- 错误提示
- 提交后的状态反馈

例子：

```text
字段：目标用户
占位提示：例如：刚开始做小红书的个人博主
说明文字：越具体，生成结果越贴近真实场景
错误提示：请填写目标用户
```

你可以这样要求 AI：

```text
请帮我设计一个 AI 文案生成表单。

要求：
1. 每个字段都有 label、placeholder 和说明文字
2. 必填字段为空时要有错误提示
3. 提交按钮要有 loading 状态
4. 表单不要太挤，字段之间保持清晰间距
5. 移动端一列展示，桌面端可以左右分栏
```

## 6. 卡片设计：卡片不是越多越好

现代应用里经常使用卡片，但新手很容易把整个页面都做成一堆卡片。

卡片适合用在这些场景：

- 展示一组重复项目
- 包住一块独立信息
- 表达一个可点击入口
- 展示统计数据

卡片不适合用来包住整个页面。页面本身不需要再放进一个大卡片里。

一个简单判断标准是：**如果这块内容可以独立移动、删除或复用，它适合做成卡片。**

例如：

```text
适合卡片：
- 一条生成历史
- 一个项目入口
- 一个价格套餐
- 一个统计指标

不适合卡片：
- 整个页面主体
- 一整段说明文字
- 所有导航和内容的外层容器
```

## 7. 空状态、加载状态和错误状态

很多教学项目只做“正常状态”，但真实产品里，用户经常遇到的是：

- 还没有数据
- 正在加载
- 请求失败
- 没有权限
- 输入内容不完整

这些状态如果没有设计，应用就会显得很粗糙。

至少要为每个核心页面设计 4 种状态：

| 状态 | 用户看到什么 | 用户能做什么 |
| --- | --- | --- |
| 默认状态 | 正常内容 | 浏览、操作 |
| 空状态 | 暂无数据说明 | 创建第一条内容 |
| 加载状态 | 骨架屏或 loading | 等待 |
| 错误状态 | 错误原因和重试入口 | 重试、返回 |

例如生成结果区：

```text
空状态：填写左侧表单并点击生成，结果会显示在这里
加载状态：正在生成文案...
成功状态：展示生成结果，提供复制和重新生成按钮
错误状态：生成失败，请检查网络或稍后重试
```

## 8. Dashboard 页面怎么设计

Dashboard 是很多全栈项目都会用到的页面，比如后台、数据看板、管理系统。

一个基础 Dashboard 可以这样组织：

```text
页面标题：数据概览

第一行：关键指标
- 今日生成次数
- 活跃用户
- 付费转化
- 失败任务

第二行：主要图表
- 趋势图
- 分类占比

第三行：最近记录
- 最近生成任务
- 最近用户反馈
```

设计 Dashboard 时，不要一上来就塞很多复杂图表。先问三个问题：

1. 这个页面服务谁？
2. 用户进来最想知道什么？
3. 哪个指标会影响下一步行动？

对于第一个作品集项目，Dashboard 不需要非常复杂，但要做到结构清楚。

## 9. 移动端适配：不要只看电脑屏幕

即使你的项目主要在电脑端展示，也应该检查移动端是否不会崩。

最基础的移动端规则：

- 页面内容改成单列
- 表格在移动端可以横向滚动
- 按钮不要太小
- 表单字段不要左右挤压
- 文字不要溢出容器
- 重要操作不要藏得太深

你可以这样要求 AI：

```text
请确保这个页面响应式适配：
1. 桌面端使用左右分栏
2. 移动端改成上下排列
3. 表单控件宽度为 100%
4. 按钮在移动端保持足够点击面积
5. 长文本不要溢出卡片
```

## 10. 给 AI 的 UI 生成提示词模板

下面是一份可以直接复用的提示词：

```text
请帮我设计并实现一个现代 Web 应用页面。

页面名称：[填写页面名称]
目标用户：[填写用户类型]
核心任务：[用户在这个页面要完成什么]

页面结构：
1. 顶部区域：包含页面标题、说明文字和主要操作按钮
2. 内容区域：根据功能拆成清晰模块
3. 反馈状态：包含空状态、加载状态、错误状态

设计要求：
1. 风格干净、专业，适合真实产品演示
2. 信息层级清楚，不要所有文字一样大
3. 一个区域内只保留一个主按钮
4. 表单字段要有 label、placeholder、说明文字和错误状态
5. 卡片只用于独立内容，不要把整个页面套进大卡片
6. 支持桌面端和移动端响应式
7. 不要使用大面积渐变背景，不要堆装饰元素
8. 生成完整可运行代码，并说明组件结构

验收标准：
1. 页面第一眼能看出用途
2. 用户知道下一步该点哪里
3. 所有状态都有对应展示
4. 移动端不出现文字溢出或按钮挤压
```

## 11. UI 自检清单

完成页面后，用下面这份清单检查：

### 页面结构

- [ ] 页面标题是否清楚
- [ ] 当前页面的核心任务是否明确
- [ ] 页面是否分成了清晰区域
- [ ] 用户是否能快速找到主操作

### 信息层级

- [ ] 标题、说明、正文的字号和字重是否有区别
- [ ] 重要信息是否比辅助信息更突出
- [ ] 页面是否有足够留白
- [ ] 是否存在过多颜色抢注意力

### 组件状态

- [ ] 按钮是否有默认、hover、disabled、loading 状态
- [ ] 表单是否有错误提示
- [ ] 列表是否有空状态
- [ ] 请求失败时是否可以重试

### 响应式

- [ ] 移动端是否能正常阅读
- [ ] 表单是否变成单列
- [ ] 表格或长内容是否处理了溢出
- [ ] 按钮是否足够容易点击

### 可演示性

- [ ] 页面是否像一个真实产品，而不是临时 Demo
- [ ] 截图放进作品集是否清楚
- [ ] 面试时能否讲清楚这个页面的设计取舍

说明可以按这个格式写：

```md

我让 AI 修改过的地方：
```

## 总结

UI 设计不是把页面装饰得更复杂，而是让用户更容易理解和操作。

对于 AI 编程学习者来说，最重要的是建立一套稳定判断标准：

- 页面结构是否清楚
- 信息层级是否明确
- 操作是否有主次
- 状态是否完整
- 移动端是否可用

当你能把这些要求清楚地告诉 AI，它生成的页面质量会明显提高。你也会从“让 AI 随机生成一个页面”，逐渐变成“像产品设计师一样指挥 AI 完成界面”。
