# 1.2 动手做出原型

在本节中，我们会从零开始，借助 AI IDE 动手搭建一个可以实际运行的 Web 应用原型。你将学会如何把脑海中抽象的产品想法，逐步拆解并落地为具体的界面与代码，同时在遇到问题时，能够有条理地排查原因并解决错误。

## 1.2.1 把需求变成代码的过程

### 学习目标

我们希望通过本节的学习，你能够建立起“从产品需求到代码实现”的完整心智模型。你需要学会如何用自然语言准确地向 AI 描述自己的想法，并掌握把复杂需求拆解成若干可执行小步骤的方法。只有打牢这部分基础，后续在真正做项目时，你才有能力独立完成从构思到实现的全流程。

### 核心内容

**需求描述的技巧**是和 AI 高效协作的关键前提。你需要学会如何清晰、准确地说出自己真正想要的效果，而不是一句“帮我做个 XXX 页面”就草草结束。模糊的描述往往会导致 AI 生成风格不统一、逻辑不清晰甚至完全不相关的代码；相反，越是具体、可执行的描述，越能帮助 AI 生成贴合你预期的结果。日常写提示词时，可以多使用明确的动词和名词，少用“好看一点”“高级一点”这类模棱两可的形容词。

**迭代式开发**是现代软件工程的基本范式，对 AI 辅助开发同样适用。我们强烈建议你遵循“从简单到复杂”的原则，不要一上来就要求 AI 把所有功能、所有页面一次性写完。更稳妥的做法是，先让最核心的流程能够跑通，做出一个可用的 MVP（最小可行性产品），再在此基础上逐步补充功能、打磨细节、优化体验。这样不仅更容易掌控项目节奏，也方便你随时根据新的想法调整方向。

**AI 辅助开发工作流**大致可以概括为“提问 → 生成 → 验证 → 修正”的循环过程。你需要在这个循环中清楚地意识到自己扮演的角色：提问时更像是产品经理，要把需求讲清楚；生成阶段是监督者，要判断 AI 给出的方案是否合理；验证阶段是测试工程师，要认真检查功能是否按预期工作；修正阶段则是技术负责人，要综合考虑如何修改更合适。熟练掌握这一套循环，可以显著提升你的整体开发效率。

### 实践任务

请选择一个简单的应用想法，例如一个待办清单或者简易笔记应用。尝试用自然语言完整地描述它，不仅仅是说“做一个待办清单”，而是要描述它的核心功能、用户如何操作以及预期的反馈。最后，将其拆解为 3-5 个具体的开发步骤，例如“第一步：创建页面结构；第二步：实现添加任务功能；第三步：实现删除任务功能”。

## 1.2.2 从一个单页面开始

### 学习目标

本节的目标，是帮助你弄清一个网页到底由哪些部分组成，以及 HTML、CSS 和 JavaScript 分别承担什么角色。你将学会使用 AI IDE 创建自己的第一个页面，并初步掌握按钮、输入框、文本区域等基础页面元素的用法，为后续搭建更复杂的界面打下基础。

### 核心内容

**HTML**（超文本标记语言）是网页的骨架，它决定了页面上“有哪些东西”。你可以把它类比为房子的钢筋混凝土结构，它定义了哪里是客厅（主体内容）、哪里是窗户（图片）、哪里是门（链接），但本身并不关心好不好看。

**CSS**（层叠样式表）可以理解为页面的装修师，它决定了元素长什么样、如何排列。通过 CSS，你可以调整文字的颜色和字号，设置背景、边框和阴影，也可以让元素整齐地排布在页面上，让原本只是“骨架”的页面变得更加美观、舒适。

**JavaScript** 则负责页面的交互逻辑，它让原本静态的页面“动”起来。当用户点击按钮时应该发生什么？当用户在输入框中输入内容、提交表单时系统该做出怎样的回应？这些行为背后，几乎都是由 JavaScript 在驱动。它像是网页的“神经系统”，赋予了网页与用户互动的能力。

**向 AI 描述页面**时，建议采用“布局 + 样式 + 交互”的结构化方式来组织你的描述。你可以先说清楚页面的整体布局，例如“顶部有导航栏，中间是内容区，底部有页脚”；接着再说明你期望的样式特征，比如“按钮是蓝色的、带圆角、整体风格简洁”；最后再补充关键的交互逻辑，例如“点击提交按钮后，在页面顶部弹出一个提示信息”。这样的分层描述能大大减少沟通成本。

### 实践任务

请使用 AI IDE 创建一个简单的单页应用。这个应用不需要太复杂，但至少要包含 3 个交互元素：例如一个用于输入信息的输入框，一个用于触发操作的确认按钮，以及一个用于显示最终结果的文本区域。试着让这些元素能够协同工作，完成一个简单的交互流程。

## 1.2.3 遇到报错了怎么办

### 学习目标

在编程过程中，报错几乎是每天都会遇到的事情。本节的目标，是帮你建立起更健康的错误处理心态，让你不再把报错当成“灾难”，而是当成推进项目前进的提示信号。你将学会识别常见的错误类型，并掌握向 AI 寻求帮助的合理方式，让 AI 真正成为你调试和排错时的可靠伙伴。

### 核心内容

**常见错误类型**大致可以分为三类。第一类是**语法错误**，就像写文章时出现了严重的错别字，导致机器根本看不懂你在说什么，这类错误通常在写代码或保存时就会被提示出来。第二类是**运行时错误**，代码在形式上没有问题，但在实际运行过程中遇到了意料之外的情况，比如访问了一个不存在的变量或文件，程序因此中断。第三类是**逻辑错误**，代码可以顺利运行也不报错，但结果却和预期不一致，例如金额计算少算了一位，这类错误往往最隐蔽，也最考验你的分析能力。

**阅读错误信息**是解决问题的第一步。不要下意识地害怕那些一大片红色的报错文字，它们其实往往已经给出了非常关键的线索。报错信息通常会告诉你错误发生在哪个文件的哪一行，以及大致的错误原因。学会从中提取关键词，再带着这些关键词去搜索或询问 AI，是快速定位问题的重要能力。

**向 AI 提问**时，可以采用一个简单实用的模板：复制完整的错误信息，附上相关的代码片段，并补充说明“你本来希望发生什么”和“实际发生了什么”。这三个要素齐全时，AI 往往就能较为准确地理解你所处的上下文环境，从而给出更有针对性的修改建议。

**调试基础**中，最常见也最实用的方法就是使用 `console.log`。你可以在代码的关键位置打印出变量的值或程序的执行状态，就像侦探在不同场景留下标记，通过这些“路标”一步步缩小问题出现的范围，最终锁定真正的根源。

### 实践任务

为了更好地掌握调试技巧，请故意修改代码制造一些错误。例如，你可以试着删掉代码中的一个括号，或者故意拼错一个变量名。观察控制台会输出什么样的报错信息，然后尝试将这些信息发给 AI，练习如何利用 AI 来修复这些错误。

## 1.2.4 做多个页面的应用

### 学习目标

从单页应用走向多页应用，是你搭建真实产品路上的一个重要台阶。本节会带你理解“页面路由”的核心概念，学会创建包含多个页面的应用结构，并掌握在不同页面之间传递数据的一些基础方式，为后续构建更复杂的应用打下扎实基础。

### 核心内容

**路由（Routing）**可以看作是多页面应用的交通枢纽，它决定了当 URL 发生变化时，浏览器应该展示哪一个页面。你可以把它理解为网站内部的导航地图，清楚地告诉浏览器“去哪里找什么内容”。

**页面跳转**是用户在应用中穿梭的方式。相比让用户手动在地址栏里修改 URL，我们更希望通过点击按钮或链接来触发页面切换，让整个过程自然、顺畅，不打断用户的使用节奏。

**状态管理**则关注的是数据在多个页面之间如何保持一致。当用户从商品列表页跳转到购物车页时，购物车里的商品不能凭空消失，这就需要有一套机制来保存和共享这些数据。状态管理的核心目标，就是保证数据在页面不断切换的过程中依然连贯、可靠。

**用户体验的连贯性**是多页面设计中不可忽视的一环。在页面跳转时，如何保持视觉风格的一致？如何尽量缩短加载等待时间？如何通过动画或过渡效果让切换显得自然、不突兀？这些细节都会直接影响用户对整个应用专业程度的感受。

### 实践任务

请将之前的单页应用扩展为一个至少包含 3 个页面的应用。例如，你可以设计一个包含“首页”、“详情页”和“设置页”的小应用。实现页面间的流畅跳转，并尝试在不同页面间传递一些简单的数据，比如在设置页修改用户名，在首页也能同步显示。

## 1.2.5 把原型做得像那么回事

### 学习目标

一个好的原型不仅要“能用”，还要“好用、好看”。本节会带你了解一些最基本的 UI/UX 原则，学会借助 AI 改进界面设计，并掌握几种让原型看起来更真实、更专业的常用手段。

### 核心内容

**配色和排版**是视觉设计的底层基石。只要能够遵循一些简单的视觉规范，保持色彩搭配和谐、排版结构清晰，就能有效避免页面给人“杂乱”“廉价”的第一印象。你可以让 AI 帮你推荐几套适合当前场景的配色方案，或者请它按照某款成熟产品的风格来生成参考样式。

**UI 组件**可以理解为搭建专业界面的积木。善用成熟的组件库（例如卡片、列表、导航栏等），既能显著节省你在样式和交互上的时间投入，又能保证整体界面的一致性和专业感。与其从零开始造轮子，不如把精力更多地放在业务逻辑和体验优化上。

**过渡和动画**则是提升交互质感的“调味剂”。适当添加一些 hover 效果（鼠标悬停时的细微变化）、按钮点击反馈或页面切换时的过渡动画，往往就能让整个应用显得更灵动、更有细节，从而给用户留下更加愉悦的使用体验。

**移动端适配**在移动互联网时代的重要性不言而喻。你的应用不应只在电脑屏幕上表现正常，还要在手机、平板等不同尺寸的设备上保持合理的布局和良好的可操作性。掌握一些基础的响应式设计方法，可以让同一套页面在不同终端上都“看得清、点得准、用得顺”。

**AI 美化**可以视作你身边随叫随到的“虚拟设计师”。你完全可以请 AI 扮演 UI 设计师的角色，把你的设计诉求翻译成具体的 CSS 代码。只要清楚地告诉它“我想要一个现代简约风格的卡片”，并补充一些细节偏好，AI 往往就能帮你生成一套可直接使用或稍加修改就能落地的样式方案。

### 实践任务

请重新审视你当前的应用原型，有意识地从“界面和体验”的角度进行一次全面体检。结合本节提到的要点，适当调整配色和排版结构，替换或补充一些更合适的 UI 组件，并为关键操作添加恰到好处的交互动画。你的目标，是让这个原型不再像课堂上的练习 Demo，而是更接近一个真实存在、可以拿去给别人使用和展示的产品。

## 总结

完成本节学习之后，你应该已经能够独立使用 AI IDE 搭建出一个包含多个页面的 Web 应用原型。当遇到错误时，你不再会因为满屏的报错信息而手足无措，而是能够冷静地分析原因，并借助 AI 高效地完成排查和修复。更重要的是，你已经掌握了让原型在“能用”的基础上进一步迈向“好用、好看”的一整套思路和方法。
